## diagnose_migrations.py

Comprehensive diagnostic tool for analyzing Alembic migration health and identifying issues.

**Features:**
- Executes `alembic history` to view the complete migration chain
- Executes `alembic current` to verify database version state
- Scans all migration files for duplicate revision IDs
- Validates Python syntax using AST parsing
- Identifies orphaned migrations without proper down_revision links
- Detects circular dependencies in migration chain
- Checks for gaps and conflicts in migration chain
- Generates a detailed audit report to `migration_audit_report.txt`

**Usage:**
```bash
python scripts/diagnose_migrations.py
```

The script will:
1. Scan all migration files in `alembic/versions/`
2. Parse each migration file to extract metadata (revision, down_revision, etc.)
3. Run comprehensive checks for common migration issues
4. Execute alembic commands to verify database state
5. Generate a detailed report at `migration_audit_report.txt`

**Exit Codes:**
- `0`: All checks passed (warnings may exist)
- `1`: Critical issues found

**Issues Detected:**
- Duplicate revision IDs across multiple files
- Syntax errors in migration files
- Orphaned migrations referencing non-existent parent revisions
- Circular dependencies in migration chain
- Missing upgrade() or downgrade() functions

**Output:**
The script generates `migration_audit_report.txt` containing:
- Summary of issues, warnings, and info messages
- Detailed list of all critical issues
- Complete migration details for each revision
- Visual migration chain graph
- Alembic history output
- Current database version state

**Example:**
```bash
$ python scripts/diagnose_migrations.py
Starting Alembic Migration Diagnostics...
======================================================================

Report generated: /path/to/migration_audit_report.txt

======================================================================
DIAGNOSTIC SUMMARY
======================================================================
Critical Issues: 3
Warnings: 2
Total Migrations: 67
```

## validate_schema.py

Validates database schema consistency with SQLAlchemy models.

**Features:**
- Compares database tables with SQLAlchemy model definitions
- Validates foreign key relationships
- Checks for missing indexes on foreign keys
- Verifies enum type existence
- Detects schema drift between models and database
- Reports missing tables and columns

**Usage:**
```bash
python scripts/validate_schema.py
```

**Exit Codes:**
- `0`: Schema validation passed
- `1`: Issues found (see output for details)

**Checks Performed:**
1. **Table Validation**
   - Tables defined in models but missing in database
   - Tables in database but not defined in models

2. **Foreign Key Validation**
   - Counts foreign keys in models vs database
   - Identifies mismatches

3. **Index Validation**
   - Verifies all foreign keys have indexes
   - Reports missing indexes (performance impact)

4. **Enum Type Validation**
   - Checks all expected enum types exist
   - Identifies missing enum types

**Example Output:**
```bash

  - volunteer_hour_logs.supervisor_teacher_id
```

**When to Run:**

## generate_clean_migration.py

Automates the workflow for generating clean autogenerated migrations when the migration chain is too broken to repair incrementally.

**Features:**
- Stamps database with current migration head
- Generates autogenerated migration using `alembic revision --autogenerate`
- Provides interactive review and editing guidance
- Suggests testing steps
- Handles the complete workflow automatically

**Usage:**
```bash
# Full workflow (stamp database + generate migration)
python scripts/generate_clean_migration.py

# Only stamp database to current head
python scripts/generate_clean_migration.py --stamp-only

# Only generate migration (don't stamp first)
python scripts/generate_clean_migration.py --generate-only

# With interactive review
python scripts/generate_clean_migration.py --review
```

**Workflow Steps:**
1. Checks Alembic installation and database connectivity
2. Stamps database to HEAD (marks current state without running migrations)
3. Generates autogenerated migration comparing models to database
4. Reviews generated migration for common issues
5. Provides editing guidance (existence checks, false positives)
6. Suggests comprehensive testing steps

**When to Use:**
- Schema drift between models and database
- Migration chain is too broken to fix incrementally
- After major model refactoring
- Need to sync database with updated models

**Exit Codes:**
- `0`: Workflow completed successfully
- `1`: Errors during workflow

**Example:**

## clean_autogenerated_migration.py

Analyzes and cleans autogenerated migration files by detecting common issues and suggesting fixes.

**Features:**
- Detects dangerous DROP operations
- Identifies missing existence checks for enums and tables
- Finds unsafe ALTER COLUMN operations
- Checks for potential data migration needs
- Suggests code improvements with examples
- Reports issues by severity (HIGH, MEDIUM, LOW)

**Usage:**
```bash
python scripts/clean_autogenerated_migration.py <migration_file>

# Example:
python scripts/clean_autogenerated_migration.py alembic/versions/041_sync_database_with_models.py
```

**Analysis Categories:**

1. **HIGH Priority Issues**
   - DROP operations (tables, columns, constraints)
   - ALTER COLUMN operations that may lose data
   - Type changes that require data migration

2. **MEDIUM Priority Issues**
   - Enum creation without existence checks
   - Table creation without idempotency checks
   - Foreign key additions
   - Data migration requirements

3. **LOW Priority Issues**
   - Index creation without existence checks
   - Missing function definitions
   - General optimization suggestions

**Example Output:**
```bash
$ python scripts/clean_autogenerated_migration.py alembic/versions/041_sync.py

Analyzing: 041_sync_database_with_models.py
```

## create_admin.py

Creates an admin user for the application.

**Prerequisites:**
- Database migrations must be run first (`alembic upgrade head`)
- Permissions and roles must be seeded (migration 002)

**Usage:**
```bash
python scripts/create_admin.py
```

The script will prompt you for:
- Admin email
- Admin username
- Admin password (minimum 8 characters)
- First name (optional, defaults to "Admin")

1. Create a default institution if it doesn't exist

## Migration Workflow

For complete workflow documentation, see:
- **`docs/AUTOGENERATED_MIGRATION_WORKFLOW.md`** - Complete step-by-step guide
- **`alembic/versions/TEMPLATE_autogenerated_migration.py`** - Code reference with examples

### Typical Workflow

1. **Detect Schema Drift**
   ```bash
   python scripts/validate_schema.py
   python scripts/diagnose_migrations.py
   ```

2. **Generate Clean Migration**
   ```bash
   python scripts/generate_clean_migration.py
   ```

3. **Analyze and Clean**
   ```bash
   python scripts/clean_autogenerated_migration.py alembic/versions/041_sync_database_with_models.py
   ```

4. **Edit Migration**
   - Review output from analysis
   - Add existence checks
   - Remove false positives
   - Add data migrations if needed

5. **Test Migration**
   ```bash
   # Test on clean database
   alembic upgrade head
   
   # Test on database copy
   # (restore backup first)
   alembic upgrade head
   
   # Test rollback
   alembic downgrade -1
   alembic upgrade head
   ```

6. **Validate Result**
   ```bash
   python scripts/validate_schema.py
   python scripts/diagnose_migrations.py
   ```

7. **Apply to Production**
   ```bash
   # Create backup first!
   pg_dump production_db > backup.sql
   
   # Apply migration
   alembic upgrade head
   ```

## Best Practices

1. **Always Test First**
   - Test on clean database
   - Test on database copy with data
   - Test rollback functionality

2. **Use Existence Checks**
   - Make migrations idempotent
   - Check before creating objects
   - Handle already-exists errors gracefully

3. **Review Autogenerated Migrations**
   - Never trust autogeneration blindly
   - Look for false positives
   - Verify DROP operations

4. **Document Everything**
   - Add clear docstrings to migrations
   - Document manual steps required
   - Note data migration needs

5. **Maintain Migration Chain**
   - Keep chain linear when possible
   - Fix conflicts immediately
   - Run diagnostics regularly
